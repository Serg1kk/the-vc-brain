# `suggest-followup-questions` — decisions & open items

Built 2026-07-19 by the feature-09 workflow build (n8n-workflow-builder agent), under delegated
authority — the operator was not available for a live per-question brainstorm gate, so this session
supplied complete answers up front instead of pausing. Sources: `lovable-brief.md` §6/§9.4/§12.5,
`data-contracts.md` §1/§4/§8, the sibling `gap-question-phraser` artifact set (`08/agents/...`), live
queries against the dev database (`api_founders.founder_score_gaps`, `claim_trust`, `events` where
`event_type='claim_contradicted'`).

## Decisions taken

| # | Decision | Rejected alternative | Why |
|---|---|---|---|
| D-1 | **Contradictions never reach this agent.** The workflow reuses `events.payload.question` verbatim and builds `why` from `founder_claim`/`found_reality`/`checked_at`, entirely in code. | Send contradictions through this agent too, for one unified phrasing pass. | Feature 05 already wrote a fully-formed, neutrally-framed question at verification time — re-phrasing it through a second model call is pure risk (a second chance to drift into accusation framing) for zero benefit. Reusing stored text is the `▦` (rule) side of this feature's chip system; only the two gap-description sources are genuinely `◇`/`▦◇` work. |
| D-2 | **One agent handles both `founder_score_gap` and `missing_claim` items**, distinguished by `kind`. | Two separate agents, one per source. | Both are the same underlying skill — turn a described-or-labelled gap into one spoken question — and splitting them would double the register-drift risk the sibling agent's own D-7 already warned about (one agent per output shape, not per data source). |
| D-3 | **Register is "prepared colleague," not "coaxing form."** Explicitly does NOT reuse `gap-question-phraser`'s Mom-Test/hide-the-evaluation framing. | Copy the founder-facing tone wholesale for consistency across the app. | The audiences are different in a way that matters: `gap-question-phraser` writes to a first-time applicant who can be scared off by feeling evaluated (measured continuation-rate cost, cited in the sibling's own TBD-3). This agent writes to an investor who already agreed to a diligence call — hiding the diligence framing here would just read as evasive, not kind. |
| D-4 | **Static per-`kind` fallback** ships alongside the agent for any item the validation gate drops, mirroring the sibling's D-4. | Ship nothing when validation fails. | A dropped question shrinks the "where to dig" set on a card the investor is about to open before a real call — silently returning fewer questions than the deterministic selection actually found is worse than a slightly generic one. Fallback text: `founder_score_gap` → *"Can you walk me through {criterion_label}, in your own words?"* / *"We don't yet have direct evidence for this."*; `missing_claim` → *"Can you tell me about {last topic segment, humanized}?"* / *"We don't have this on file yet."* |
| D-5 | **Forbidden vocabulary re-checked in code, not just in the prompt.** Same two-layer enforcement as the sibling agent's D-3. | Prompt-only enforcement. | This output reaches the investor's screen with no human review step (`input-spec.md`), same structural risk as feature 08's case, even though the audience-harm profile differs (an investor seeing leaked internal jargon looks unpolished; a founder hearing an accusation is the harm this restriction actually guards against). |
| D-6 | **Length caps 160 / 140**, slightly looser than the sibling's 140 / 120. | Reuse the sibling's exact caps. | This question is read from a card mid-call, not typed into a fixed-width form textarea — a little more room is available, and a two-clause "who, and how did they find you" style question (explicitly allowed, restrictions section) needs it. |
| D-7 | **`ref_id` is an opaque string the caller defines**, not a `criterion_id` enum like the sibling's `L2`/`L3`/`X5`. | Reuse `criterion_id` as the correlation key. | This agent's input mixes two source kinds (`founder_score_gap` and `missing_claim`), which don't share an id space — `missing_claim` items are keyed by `claim_id`, not a criterion code. An opaque, caller-defined `ref_id` keeps this agent decoupled from both sources' internal id schemes. |
| D-8 | `temperature` omitted (not `0`), matching the sibling and cross-feature rule 7 (`03/done.md`). | `temperature: 0` for reproducibility. | `gpt-5.6-luna` rejects `temperature: 0` with HTTP 400; although this agent runs on `terra`, keeping the parameter convention identical across every OpenAI-calling node in this repo prevents it being copy-pasted into a `luna` node later with a poisoned parameter. |
