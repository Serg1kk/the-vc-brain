## Decisions taken (with rationale)

| # | Decision | Rationale | Source |
|---|---|---|---|
| D1 | **The thesis appears in no prompt, no input variable, and no hint.** Only two user variables exist: `gate_text` and `structured_hints`. The input spec adds an explicit prohibition on routing thesis material (`config.geos`, mandate sectors, prior verdicts) through `structured_hints`. | D-02: «our thesis is X, does this fit?» makes a model «highly likely to hallucinate alignment even if the evidence is weak». The hint channel is the one place the thesis could re-enter while leaving the prompt technically clean, so it needed naming. | D-02, §6.1 |
| D2 | Output is a **flat** object with `reasoning` first — **not** 04's `status`/`data` envelope. | §4 fixes the shape and the plan's acceptance criterion is «`reasoning` first in the schema». An envelope would demote `reasoning` below `status`, and field order is stated to be load-bearing. The divergence from 04 is deliberate, not an oversight. | §4; plan C1 |
| D3 | **No status/error code, and no error branch in the schema.** A failed or contentless extraction is the all-null record. | An all-null record flows through the gate correctly on its own: coverage → 0 → `insufficient_evidence` → unranked, routed to enrichment (D-07). An error code would need its own handling for no behavioural gain. | D-07, §2 |
| D4 | `quotes` carries **all five keys always**, `null` where the attribute is `null`. | Strict structured output cannot express «required iff». Requiring all five unconditionally is the strongest structural form available under §4's fixed flat-map shape, and it makes the downstream pairing check a trivial biconditional rather than a presence test. | §4, §5.4 |
| D5 | A quote must be a **contiguous verbatim substring of `gate_text` or of a `structured_hints` value**, checked mechanically downstream. | `claims.text_verbatim` is NOT NULL, so a fabricated span is persisted as sourced evidence — the most damaging failure available to this agent. §1.1 defines `_text` as «the concatenated gate input», which gives the check a precise, already-normative surface to run against. | §1.1, §5.4 |
| D6 | **The «If evidence is insufficient, reason cautiously but still decide» directive is excluded**, in that wording and in every paraphrase. The prompt says the opposite: unquotable ⇒ `null` + `missing_fields`, and «completeness is not a goal; groundedness is the only goal». | §4 states it is «backwards for an extractor — it would push the model to guess sector and geography when the text states neither», and assigns it to feature 06 where a decision is mandatory. | §4 |
| D7 | `other` (sector) is a **determination**: it carries a quote and does **not** enter `missing_fields`. `null` is the gap. | The two produce different rule outcomes and must not be conflated: observed `other` → `no_match` (a real miss); `null` → three-valued `unknown` (D-03), which can never fire a hard rule. Collapsing them would let a thin deck be auto-rejected. | §1.1, D-03 |
| D8 | The model is instructed **not** to use the literal `business_model: "unknown"` to mean «could not tell»; that is `null` + `missing_fields`. | See Q1 — as a compared *value*, `"unknown"` yields `no_match` where D-03 requires `unknown`. The prompt cannot forbid a value §1.1 lists as legal, so it steers away from it and the evaluator guards it. | §1.1, D-03 |
| D9 | An **irreconcilable hint-vs-text conflict resolves to `null` + `missing_fields`**, never to either side. | §6.1: a `contradicted` claim is treated as `unknown` because «a contradicted attribute is precisely *we do not reliably know this*». Filing either side as fact would be fabrication (invariant #5). | §6.1, invariant #5 |
| D10 | The agent **must not emit `geography_region` or `stage`**; both are derived in `lib/f07/vocabulary.js`. | Two sources of truth for a derived field is exactly how rev.2 came to carry geography in three incompatible forms and soft-fail a German B2B company on its own default thesis. | §1.1, rev.2 audit |
| D11 | Push-back is instructed explicitly in two forms: **self-label vs. described product**, and **hint vs. text**. Desired Example 1 exercises the first; Example 3 the second. | §4 «Adopted: the explicit push-back instruction». An extractor that defers to a confident header would file the deck's marketing as fact. | §4 |
| D12 | **Negative criteria are enumerated per field**, alongside positive ones — four named traps: ccTLD → country, product noun → `b2b`, waitlist → `early_revenue`, using AI → `ai-infra`. | §4 «Adopted: negative criteria alongside positive ones». These are the inferences that feel correct and are unsupported; each would otherwise become a permanent sourced claim. | §4 |
| D13 | `temperature = 0`; strict JSON Schema enforced at the API. | §4 mandates both. Beyond determinism per se: a stochastic `sector` would make the verdict irreproducible for identical input, which would make `input_fingerprint` (§5.1) useless as a retry key. | §4, §5.1 |
| D14 | **Identity exclusion is a prompt restriction, not a claim of anonymization.** No name, employer, school, accelerator or personal characteristic may influence a value or appear in the output. | §4's corrected claim: the input *does* contain founder names and the model *does* ingest them, so entity-memorization bias is **not** mitigated. What is true is that the output schema has no identity field and no identity attribute can influence a rule. Input-side NER masking is post-MVP. | §4 |
| D15 | `reasoning_effort = low`, with a named escalation trigger rather than a bigger model by default. | The reasoning procedure is externalised into `<chain_of_thoughts>` and the required `reasoning` field — persisted on `ai_runs` and QA-inspectable, unlike hidden reasoning tokens billed at output rates. | model-recommendations |
| D16 | `application_id` is **not** passed to the model. | No extractable signal; an opaque identifier in a prompt is one more surface to pattern-match on. It stays in the n8n item for the write path. | input spec |

## Open questions

| # | Question | Why it is open | Disposition |
|---|---|---|---|
| **Q1** | **`business_model: "unknown"` is a legal §1.1 value that breaks D-03 if compared as a value.** A rule `{field:"business_model", op:"eq", value:"b2b"}` against observed `"unknown"` evaluates to `no_match` → outcome `missed` → the weight enters `total` but not `earned`, **lowering fit because the model could not tell**. D-03 requires `unknown` in exactly this situation. | §1.1 lists `unknown` in the closed set; D-03 defines unknown-ness by *absence* (`absent, null, in missing_fields, or contradicted`) and does not mention a sentinel *value*. The two are consistent only if something treats the sentinel as absence. | **→ Stage B (task B1, `lib/f07/vocabulary.js`).** Recommended: treat the literal `"unknown"` on `business_model` as an unknown-valued field — one line, alongside the `scaling → no mapping` rule that already does the same job for `stage`. C1 mitigated the prompt side (D8) and documented the guard in the schema `description`, but **cannot fix the evaluator from this terminal**. If Stage B declines, this is a live REQ-003-shaped defect. |
| **Q2** | **The deterministic validator node has no plan task.** Stage D's D1 acceptance criteria list the write path and the select-by-hash rule but do not name a post-LLM validator, and the pairing + substring invariants have no other enforcer. | Strict structured outputs cannot express the conditional (`if`/`then` unsupported), so the schema genuinely cannot carry it. C1 specified the four checks fully in the input spec. | **→ Stage D.** Treat the input spec's validator table as part of D1. Wiring the LLM node straight into the `claims` writer would let an ungrounded value reach a NOT NULL `text_verbatim`. Cheap — four checks, one Code node. |
| Q3 | **Quote-substring matching: exact or whitespace-normalised?** Deck-extracted text carries line breaks mid-sentence, so an honest quote may not be a byte-exact substring. | Not addressed in design.md — it did not previously specify a substring check at all. | Recommended: **normalise runs of whitespace on both sides before comparison**, and store the model's quote as returned. Exact matching would reject correct quotes for cosmetic reasons; anything looser (fuzzy, token-overlap) reopens the fabrication hole. Confirm with Stage D. |
| Q4 | **Should `what_is_built` require a quote at all?** §1.1 marks it «not gateable», so it never reaches a rule. | It is nonetheless persisted as `company.what_is_built` (§5.4.1), and `claims.text_verbatim` is NOT NULL for that row like any other. | **Resolved as required** — the NOT NULL column decides it. Recorded because a reviewer reading only §1.1's «not gateable» column might read the quote as unnecessary. |
| Q5 | **Non-English input: `what_is_built` in English, quotes in the source language.** | CLAUDE.md's language policy makes the artifact vocabulary English; verbatim-ness makes translation illegal for quotes. Design.md does not address the mix. | Decided as stated (input spec) and flagged for the memo agent: feature 06 will render an English summary beside a possibly non-English `text_verbatim`. Worth a UI note in 09. |
| Q6 | **`gate_text` cap set at ~24,000 characters.** design.md gives no cap. | Sized so a maximal deck stays well inside the 272K surcharge threshold while keeping the head, where all five attributes live; deck tails are team bios, projections and logo walls. It is a judgement, not a sourced number. | Leave open. Raise the cap before changing models if QA finds attributes in truncated tails — input is $1/1M against 1.05M of context, so the cap is cheap to relax. |
| Q7 | **Whether prompt caching is enabled.** | Payback arithmetic is favourable (repaid by the first hit inside the TTL, since the static prompt is 82% of a typical call), but total feature spend is ~$1.10 and a 30-minute TTL does not survive a once-live demo run. | Recommended **off for MVP**, matching 04's D9 — but for an operational reason, not an economic one. If the provider applies prefix caching automatically at no write premium, take it and change nothing. |
| Q8 | **`ai_runs.prompt_version` value for this agent.** Referenced by §5.4 (the `ai_runs` key fields and the `raw_signals` hash) but never assigned a value anywhere. | §5.4 makes `prompt_version` part of `sha256(application_id ‖ input_text_hash ‖ prompt_version)`, so changing it deliberately invalidates retry dedup — meaning it needs a stable, agreed value, not an ad-hoc string. | Suggest `f07-extract-v1`, bumped only when this prompt file changes materially. **→ Stage D** to confirm and use consistently; QA item 10 (idempotency) depends on it being stable across retries. |

## Alternatives considered and rejected

| Alternative | Why rejected |
|---|---|
| **Put the thesis in the prompt and ask «does this fit?»** — what all 20 OSS clones do | The sycophancy trap D-02 exists to close: the model hallucinates alignment on weak evidence. It also destroys reuse — attributes are thesis-independent and extracted once, then evaluated against any number of theses. |
| Pass a *narrowed* hint instead of the full thesis («we care about sector and geography») | Same defect in weaker clothing: it tells the model which fields the answer hinges on, which is precisely the pressure to fill them. The prohibition in the input spec covers this form explicitly. |
| Include «If evidence is insufficient, reason cautiously but still decide» | §4 rejects it by name for this agent: it would push the model to guess sector and geography when the text states neither. It belongs to feature 06, where a decision is mandatory. |
| Let the model emit a per-field `confidence` score | Phillips et al. (cited in §3.2): model-internal confidence is unreliable — the «confidently wrong regime» — so a self-reported `0.87` is unusable. Confidence in this feature is *coverage*, computed from rule weights (D-07), not asked for. |
| `{value, quote}` object per attribute, so the schema could require them together | Deviates from §4's fixed flat `quotes` map, which is the shape the write path and the reviewers work from. The pairing is enforced by the validator instead — one node, no contract change. |
| Skip `quotes`, let the write path use the normalized label as `text_verbatim` | A normalized label is not a verbatim span. It would satisfy the NOT NULL column while making every `claims` row self-referential, silently destroying the evidence trail that Trust Score (invariant #3) is built on. |
| Emit `geography_region` directly and skip `region_of(country)` | Two sources of truth for a derived value — the rev.2 defect that made a German B2B company soft-fail the shipped thesis's geography rule. |
| Free-text `sector` instead of a closed set | The rules in `theses.config` compare against fixed operand arrays (`expr.value`), so a free-text sector would `no_match` almost everything and quietly depress fit. The closed set *is* the contract. |
| One combined agent doing extraction **and** rule evaluation | Collapses D-02 and re-imports the sycophancy trap, pays LLM prices for arithmetic that is deterministic, and makes the gate irreproducible — killing `input_fingerprint` (§5.1) and the «model proposes, backend decides» pattern this project already uses. |
| A bigger model (Terra/Sol) as the answer to inference errors | The failure mode is inferences that *feel* correct (ccTLD → country), which is a prompt-and-validator problem. Terra costs 2.5× and leaves the negative criteria, the quote requirement and the substring check doing the same work. Escalation trigger is named in the model recommendations instead. |
| A terse prompt to save the 6,800 static tokens | Saves ~$0.004 per application against ~$1.10 of total feature spend, and gives up the enumerated negative criteria — the only defence against four inferences that each become a permanent sourced claim. Optimising the wrong number by two orders of magnitude. |
