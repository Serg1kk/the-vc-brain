## Данность (не обсуждается)

- **API — обязателен** (требование оператора): сервис = сайт + API; агент подключается через
  CLI (позже возможно MCP) — получать, обновлять, искать.
- Инварианты спонсора: REQ-001..011 (`internal/Meetings/requirements.md`), скоуп-вердикты SCOPE-001..008.
- Время: ~10-12 рабочих часов соло + **1.5-2ч на сабмишен-артефакты** (3 видео + summary + zip).

## Персоны (MVP)

| # | Персона | Что делает в продукте | Ключевые функции |
|---|---|---|---|
| P1 | **Инвест-менеджер** (нетехнический, primary — REQ-007) | Настраивает тезис фонда, смотрит ranked-фид, открывает карточку фаундера, читает memo, решает | Thesis config · dashboard · founder card (оси+evidence) · memo · suggested deep-dive questions |
| P2 | **Агент/автоматизация** (primary наравне с P1 — наша фишка) | Подключается по API/CLI: заливает кандидатов, читает скоры+evidence, ищет NL-запросом, следит за watchlist | REST API · CLI · NL-search · webhooks/watchlist (opt) · MCP (stretch) |
| P3 | **Фаундер-заявитель** (secondary) | Подаёт минимальную заявку (deck + name + ссылки на артефакты), видит статус, может удалить данные | Intake form · status · **opt-out (GDPR)** |
| — | *Судья хакатона* (design-constraint, не персона) | За 3 минуты видит весь путь signal→decision | Демо-сценарий ниже |

## Варианты функционального скоупа

### Вариант A — «Vertical Slice: Signal → Decision» ⭐ рекомендую
Один живой путь целиком, на 3-5 РЕАЛЬНЫХ фаундерах: сигнал/заявка → cold-start Founder Score →
3 оси + per-claim Trust → memo → $100K-рекомендация. Sourcing внутри слайса — **один канал глубоко**
(GitHub-хаб → HN → личный сайт, identity resolution без ML). Дашборд минимальный, но investor-grade.

- ЗА: Investment Utility 30% (инвестор реально может действовать), Data 30% (глубина канала + evidence),
  совпадает с REC-001/REC-002 (Карл) и советом Opus-агента; демо рассказывается одной историей.
- ПРОТИВ: узкий сорсинг-фронт (1 канал), меньше «вау-ширины».

### Вариант B — «Sourcing-Heavy: Founder Radar»
Ядро — радар: непрерывный скан GitHub/HN → discovery-фид «founders you should know» → cold-start
скоринг + watchlist (Memory). Diligence/memo — тонкие (score breakdown + evidence + gaps, memo-lite).

- ЗА: максимальная ставка на слова Карла «sourcing = наименьшая конкуренция»; Data 30%.
- ПРОТИВ: рискуем Investment Utility 30% — memo-lite может не дотянуть до «decision-ready»;
  required-секции memo (REQ из брифа) просядут.

### Вариант C — «Full Pipeline Thin» ❌ не рекомендую
Все 4 стадии + Thesis Engine + оба трека + committee. Ровно анти-паттерн RSK-001
(«AI-обёртка», 4 полусделанных модуля) — включён для полноты картины.

**Рекомендация: A, с B-сердцем.** Vertical slice, где сорсинг-канал один, но сделан глубоко, и
глубина = наш cold-start скоринг (27 сигналов, митигации RSK-002..004). A и B сливаются почти без
потерь, если не строить два канала.

## Функциональная карта (вариант A+)

1. **Thesis Engine (конфиг)** — sectors / stage / geo / check size / risk. Конфигурируемый
   (инвариант #6), простая форма. Фильтрует и окрашивает всё ниже.
2. **Intake (inbound)** — форма: deck + company name (минимум по брифу) + опциональные ссылки на
   артефакты (repo, live URL, demo) — «артефакты > слайды» (REC-014). Двухступенчатый гейт:
   дешёвый thesis pre-filter → полный скоринг (REC-008, SCOPE-007).
3. **Sourcing Radar (outbound, 1 канал глубоко)** — GitHub-профиль как хаб → HN → личный сайт;
   discovery-фид; «suggested outreach» карточка-заглушка (STUB-001). Оба трека → одна воронка.
4. **Founder Score (ядро)** — cold-start ветка; сигналы: shipped-vs-built, provenance-check,
   agency/доведение, домен-экспертиза 40+, GTM; анти-сигналы: GitHub-звёзды, headcount, полиш
   питча. Скор персистентен (Memory), с трендом, никогда не сбрасывается (REQ спонсора).
5. **3 оси + Trust** — Founder / Market / Idea-vs-Market раздельно (REQ-002) + per-claim Trust
   (REQ-003: missing → confidence вниз). Validator-агент (GVC-паттерн, REC-007) — stretch.
6. **Memo + Decision** — required-секции: snapshot, hypotheses, SWOT, problem & product,
   traction; gaps помечены честно (REQ-004); suggested deep-dive questions для звонка (REC-005);
   $100K-рекомендация с условиями.
7. **Experience (P1)** — dashboard (ranked + trend) → founder card (score breakdown + evidence
   ledger) → memo view. Notion-approachability.
8. **API-first (P2)** — REST: submit / get score+evidence / get memo / **NL-search**
   («technical founder, Berlin, AI infra, no prior VC backing» одним запросом — REQ брифа) + CLI-обёртка.
   MCP — stretch.
9. **Data Ethics слой** — только публичные данные, opt-out кнопка, data-minimisation
   (план подачи из `internal/research/data-sources.md`) — дёшево, судьям заходит.

## Что честно заглушаем (по intel-базе)

Реальный outreach (draft-карточка) · reference calls (прокси/«unavailable») · financials/cap table
(«not disclosed») · investment committee и соцсети-personality (stretch) · downstream — не делаем.

## Демо-сценарий (3 мин, для видео и финала)

Thesis config → радар нашёл реального фаундера (никого нет в Crunchbase!) → score с evidence по
осям → NL-search → memo с gaps и deep-dive вопросами → «инвестор действует». Финальный слайд:
**«3 наивные ошибки, которые мы НЕ сделали»** (вайбкод-деградация / LLM-эхо-камера / survivorship
bias — RSK-002..004 с первоисточниками).

## РЕШЕНИЯ (оператор, 18.07 ~23:40)

1. **Скоуп: вариант A+** (вертикальный слайс, один глубокий канал сорсинга) + **мульти-канальный
   дашборд**: нереализованные каналы — честные заглушки, UI показывает полное видение.
2. **Главный контекст P1: K1+K2** (разбор потока → подготовка к звонку). Радар K3 — на одном
   канале, остальные каналы заглушками.
3. **Founder-сторона: КОМПАКТНЫЙ интейк (вариант Б — ПЕРЕСМОТРЕНО 19.07 ~02:00, заменяет
   «полный цикл»).** Перепроверено против речи и документа самого Карла: двусторонность —
   дословно его («Fix работает в обе стороны», метафора заявки-кредитки, «apply through a
   form and get through that door») — но ЕГО вход это минимальная форма («should not get too
   long»), а весь бальный вес рубрики на инвесторской стороне. Итого:
   - **Минимальная форма** (deck + название + ссылки на артефакты) + **загрузка ЛЮБЫХ
     файлов/презентаций** — deck парсим (извлечение текста → claims); прочие типы файлов
     опционально, **для MVP допустима dummy-кнопка загрузки** (честная пометка «parsed later»).
   - **Опциональный короткий агент-опрос**: 3-5 вопросов из gaps (текстовый чат), pre-fill
     из публичных следов; можно пропустить — никогда не стена. Гардрейлы из ресерча остаются.
   - **Голосовой ввод/TTS — stretch внутри MVP** (только если останется время), не ядро.
   - **Follow-up упрощён до «дополнительных вопросов» по инициативе менеджера:** VC-менеджер
     оставляет пометки/комментарии на карточке («чекнуть», «проверить») → на основе пометок +
     gaps AI ПРЕДЛАГАЕТ форму доп-вопросов для отправки фаундеру; отправка/email — красивая
     ЗАГЛУШКА (показываем собранную форму + письмо, ничего не шлём). Полный цикл follow-up
     интервью + голосовой агент → post-MVP.
   - **ОБЯЗАТЕЛЬНЫЙ UI-тизер** будущего AI Interview: краткий питч, почему это круто —
     отдельные скоры из интервью, метрики сравнения голос-с-записи vs живой звонок,
     сигналы интонаций/пауз (anti-gaming).
   - Освободившееся время (~1.5-2ч) → в глубину фич 04/05/06 (там 55%+ рубрики).
4. **Демо-данные:** 3-5 реальных фаундеров (HN Show HN / GitHub) + 1-2 синтетики с зашитыми
   противоречиями (безопасно показать truth-gap / red flags).
5. **API: REST + CLI + готовый Claude-скилл для CLI** — скилл с полным описанием: как работать
   с CLI, структура базы, методы, паттерны запросов; агент фонда подключает токен своей
   системы и работает. MCP — не в MVP.
6. **Техстек — РЕШЕНИЕ ОПЕРАТОРА (19.07): гибрид n8n + Supabase.** Все workflow, автоматизации,
   ассистенты и агенты продукта — **визуальные n8n-workflow** (видно и понятно, не кодом);
   backend-код — тонкий слой там, где n8n неудобен. База — **Supabase** (Postgres; PostgREST
   даёт REST из коробки; storage для голосовых артефактов). Векторной базы пока нет (оператор
   скажет, если понадобится). Фронт — SPA (эксперимент Lovable / Claude Design) поверх
   Supabase REST + n8n webhooks. CLI + Claude-скилл — поверх того же REST. docker-compose
   локально (знакомый оператору self-hosted n8n+Supabase), VPS в конце. n8n-работа ТОЛЬКО
   через глобальных агентов оператора: `n8n-requirements-orchestrator` → `n8n-workflow-builder`.
   *(Раннее сравнение архитектора FastAPI+SQLite — как справка; ключевой борроу — паттерн
   vantage «model proposes, backend decides» + append-only версионируемые скоры — переносится
   как схема скоринга внутрь n8n/Supabase.)*

### Гардрейлы founder-интервью (из Exa-ресерча 18.07 — обязательные)

Ресерч: Greenhouse 2026 (2950 соискателей), Recruiting Tech Reviews 2026 (2587 + 614 baseline),
пилот Zapier Ezra, полевой эксперимент (3000+ заявителей).

- **Дисклоужер сверху**: «тебя интервьюирует AI-агент; вот что оцениваем; ~10-15 мин; человек-
  инвестор ревьюит всё перед решением». Дисклоужер + human-опция = +1.4 балла опыта, re-apply +80пп.
- **Вопросы ТОЛЬКО из gaps карточки** — generic-вопросы = убийца completion №1; персональные
  вопросы — наш дифференциатор («не те же 15 вопросов заново»).
- **Pre-fill до вопросов** — агент приходит со знанием публичных следов; фаундер подтверждает,
  а не перепечатывает. Короче интервью → меньше 28% mid-process дропа chat-AI.
- **Ответы = self-reported claims** с низким базовым confidence, верифицируются truth-gap'ом;
  интервью НЕ скорит красноречие (SIG-018; 22% кандидатов уже используют LLM на живых интервью).
  Бонус: фрод избегает завершать интервью (Zapier: 5% флагов) — интервью = фрод-фильтр.
- **Outcome сразу** после завершения (статус + сроки) — 51% прошедших AI-интервью сегодня не
  получают ответа; тишина — жалоба №1. У нас ответ за 24ч by design.
- **Длительность заявлена заранее + прогресс-бар.**
- **Кнопка «поговорить с человеком»** (можно заглушкой) — 46% хотят опцию; большинство не жмёт.
- ⚠️ **Известный риск:** асинхронные интервью отпугивают (-50% продолжения в полевом
  эксперименте, сильнее у женщин) — митигации: pre-fill (коротко), дисклоужер, human-опция,
  интервью — опциональный шаг после формы, не стена.

## Post-MVP (parking lot — оператор, 19.07)

- **Голосовой интервью-агент** (замена текстового чата в интервью): современные голосовые
  модели слышат интонации, паузы, задержки, «ищет/читает ли при ответе» — естественная защита
  от LLM-ассистированных ответов, которую текстовый чат не ловит. Отдельная задача:
  **детальный анализ рисков «текстовый агент vs голосовой»** (устойчивость к gaming,
  отпугивание/drop-off, доступность, bias голосовых суждений, юр./consent на анализ голоса).
  Явно вне 24ч-скоупа; на UI MVP — короткая пометка «Voice conversations — next phase» без
  подробностей (оператор, 19.07: *почему* — здесь, а не на экране фаундера).

  > ### 📣 ПИТЧ-МАТЕРИАЛ — зачем голос, в трёх строках (оператор, 19.07)
  >
  > Сохранено дословно для питча инвесторам. Это аргумент, а не текст для UI; на продуктовой
  > странице стоит только «coming soon».
  >
  > - **Произнесённый ответ подделать намного труднее, чем вставленный текст**, поэтому он
  >   считается более сильным свидетельством — сам артефакт несёт провенанс.
  > - **Заминки, темп и задержки несут сигнал, недоступный письменному ответу** — именно это
  >   делает голосовой ответ устойчивым к генерации на лету. Текст не отличает фаундера,
  >   который знает свои цифры, от того, кто читает их со второго экрана.
  > - **Голосовые ответы скорятся отдельно** и показываются инвестору *рядом* — никогда не
  >   смешиваясь — с доказательствами, которые мы собрали сами. Вторая независимая ось, а не
  >   более громкая версия той же самой.
  >
  > Почему это настоящий дифференциатор, а не пункт в списке фич: весь продукт построен вокруг
  > обесценивания самозаявленного сигнала (AI пишет идеальные деки, LLM отвечают на письменные
  > вопросы, убедительность девальвирована — SIG-018, FACT-014). Голос — единственный канал
  > входа, где это обесценивание ещё не произошло.
- Реальная email-доставка второго интервью; MCP-сервер; остальные sourcing-каналы из
  дашборда-заглушек.
